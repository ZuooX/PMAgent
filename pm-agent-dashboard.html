如何使用 PM Agent 工作流

在 Cursor 中开启新对话，直接用自然语言描述你要做的事，AI 会自动路由到对应阶段。
总调度规则 pm-router.mdc 始终生效，无需手动激活。

常用启动指令

"新需求 TC 4.16，H5 首页增加 AI 推荐模块" → 加载上下文 → 需求对齐
"需求对齐" → 进入目标/约束/边界对齐
"场景推演" → 用户场景 + 功能列表 + AC
"详细设计" → 页面交互 + 状态机 + 异常
"出原型" → 生成可交互 HTML 原型稿
"出需求稿" → 逐章节组装完整 PRD
"预览需求" → 构建可视化 PRD 预览（含迭代切换）
"需求评审" → 7 维度结构化自检
"修改需求" → 变更同步（改稿 + 自动同步关联产物）
"生成汇报图" → 生成迭代信息图，可截图发给市场/运营/领导

自然语言也能触发

"TC 4.11 的 H5 商详页要加一个功能"
→ 识别为 1-N 迭代，加载 TC 4.11 上下文
"我想做一个购物车功能，先聊聊目标"
→ 识别为需求对齐阶段
"梳理一下用户从下单到收货的完整路径"
→ 识别为场景推演阶段
"商详页收藏按钮的登录拦截状态机"
→ 识别为详细设计阶段
"把这次需求的所有内容整理成完整文档"
→ 识别为需求稿输出阶段
"帮我检查一下 PRD 是否有遗漏"
→ 识别为需求评审阶段

典型的完整流程示例

1
开新对话，说"新需求 TC 4.16，做一个购物车功能"
AI 自动加载项目上下文 → 进入需求对齐，逐步收集目标、约束、边界
2
对齐完成后，AI 提示"是否进入场景推演？"，你确认
围绕用户角色推演场景，提取功能列表，为每个功能写验收标准
3
功能确认后，AI 提示"是否进入详细设计？"，逐页面展开
每个页面独立输出交互 + 状态机 + 异常处理，确认一个再做下一个
4
详设完成后，说"出原型"，生成可交互 HTML
基于详细设计自动生成原型稿，输出到 output/TC-4.16/ 目录
5
说"出需求稿"，逐章节组装为完整 PRD
按模板骨架逐章输出，每章确认后再输出下一章，防止 AI 压缩
5+
说"预览需求"，生成可视化 PRD 预览
构建独立 HTML 预览页，支持迭代切换、图片灯箱、目录导航，在 Chrome 中打开
6
说"需求评审"，AI 做 7 维度自检
输出评审报告，列出必须修复项和建议优化项
7
说"修改需求"，AI 执行变更 + 自动同步关联产物
修改需求稿后，自动检查并同步 changelog、预览稿、交互稿、截图等 8 项关联产物
8
说"生成汇报图"，输出迭代信息图
从需求稿提炼核心信息，生成可截图发送给市场/运营/领导的可视化 HTML 信息图

新项目如何开始

第一次使用 PM Agent 接入一个新项目时，按以下步骤初始化。AI 会自动识别为 0-1 项目并引导你完成。

Step-by-Step

1
创建项目目录
在工作区根目录下创建项目文件夹，放入已有的背景资料
projects/你的项目名/
docs/ ← 历史需求、战略文档、竞品截图等
2
开新对话，告诉 AI 项目名
AI 检测到 project-context.md 不存在，自动进入 0-1 初始化流程
"新项目 my-saas，做一个 SaaS 订阅管理平台"
3
AI 引导你完成上下文初始化
AI 会逐步收集：产品定位、目标用户、技术栈、已有资产、竞品参考等
完成后自动生成 project-context.md
4
开始第一个迭代
上下文就绪后，直接进入正常工作流（需求对齐 → 场景推演 → ...）
"开始第一个迭代 Sprint-1，做用户注册和订阅购买"

AI 自动生成的文件结构

projects/my-saas/
project-context.md ← AI 生成：产品定位、技术栈、约束
feature-inventory.md ← AI 生成：功能清单基线
docs/ ← 你提供的背景资料
sprints/
Sprint-1/
requirement.md ← 需求稿（逐阶段产出）
changelog.md ← 变更日志
previews/ ← PRD 预览 HTML

有背景资料更好

+ 历史需求文档 → 放入 docs/，AI 自动提取
+ 竞品截图 / 分析文档 → 帮助竞品分析阶段
+ 技术架构说明 → 帮助 AI 理解技术约束
○ 没有也没关系，AI 会引导你从头梳理
后续迭代（1-N 模式）
项目初始化只需做一次。后续迭代直接说"新需求 Sprint-2，做 XX 功能"，AI 自动加载已有上下文和功能清单，进入增量模式。

跳阶段使用

不一定要从头走完整流程。你可以直接说"详细设计"跳到 Phase 3，AI 会尝试基于已有上下文工作。

多轮对话

每个阶段可以多轮对话反复打磨。AI 会在同一阶段内保持上下文，直到你确认进入下一阶段。

新对话 vs 续对话

长对话可能导致 AI 遗忘早期细节。建议每 2-3 个阶段开一个新对话，AI 会通过读取文件恢复上下文。

反压缩策略

  • ✕ 禁止一次性合并输出全文
  • ✕ 禁止"同上""略""等等"
  • ✓ 逐页面/逐章节独立输出
  • ✓ 已确认内容锁定，不可修改
  • ✓ 状态机必须穷举分支

0-1 vs 1-N 差异

  • 0-1 新项目：全量设计，从零创建上下文
  • 1-N 迭代：增量推演，读取功能清单
  • 自动识别：检测 project-context.md 是否存在
  • 增量标记：🆕新增 ✏️修改 🗑️移除 ➡️不变

核心输出物

  • 文档 project-context.md
  • 文档 feature-inventory.md
  • 文档 requirement.md (完整 PRD)
  • 原型 可交互 HTML 原型稿
  • 报告 评审检查报告

pm-alignment.mdc
需求对齐与竞品分析 · Phase 1 & 1.5

激活条件

用户开始新需求讨论、提及需求背景/目标/约束/竞品

核心行为

漏斗式收敛：业务目标 → 技术约束 → 依赖资产 → 竞品参考。AI 不主动扩大范围，帮助收敛边界。

输出检查清单

  • □ 迭代目标（一句话）
  • □ 业务驱动力 + 成功指标
  • □ 技术红线 + 时间资源约束
  • □ 平台覆盖 + 依赖项
  • □ 需求范围（做/不做）

关联模板

requirement-background.tpl.md
2
pm-scenarios.mdc
场景推演与功能列表 · Phase 2 & 2.5

核心方法

  • 1. 梳理用户角色
  • 2. 按 User Journey 推演场景
  • 3. 从场景提取功能列表
  • 4. 每个功能附带 AC（Given-When-Then）
  • 5. 定义成功度量指标

AC 编写示例

给定：用户未登录
当：点击收藏按钮
则：弹出登录弹窗，登录后自动收藏

pm-detail-design.mdc
详细设计与技术对齐 · Phase 3 & 3.5

反压缩协议

  • • 一次只处理一个页面/模块
  • • 已确认内容标记为「已锁定」
  • • 禁止省略号/同上/类似处理
  • • 状态机每个分支独立完整描述

状态机必须覆盖

正常流程 网络超时 权限不足 校验失败 并发冲突 跨端转移

页面设计内容

  • • 页面概述（路径/入口/角色）
  • • 页面结构（分区/组件）
  • • 数据来源与排序（API/配置/排序规则/加载策略）
  • • 交互设计（触发→行为→反馈）
  • • 状态机（当前态→事件→目标态）
  • • 数据需求（字段/类型/校验）
  • • 异常与边界
  • • 技术对齐清单

pm-prd-output.mdc
需求稿合并输出 · Phase 5

流水线组装策略

AI 负责生成局部，物理拼接保证无损。逐章节输出，每章独立确认。

章节输出顺序

  1. 改动日志
  2. 功能列表
  3. 需求背景与目标
  4. 需求概述
  5. 详细设计（逐页面拆分）
  6. 原型稿（HTML + 截图）
  7. 参考文档

绝对禁止

  • ✕ 对已有内容做总结/压缩/精简
  • ✕ 合并输出多个章节
  • ✕ 删减已确认的状态机/字段表格
  • ✕ 使用省略号或"同上"

6
pm-review-checklist.mdc
需求评审检查 · Phase 5.5 + 变更同步 · Phase 6

7 个检查维度

  • 1. 结构完整性（8 项）
  • 2. 功能覆盖度
  • 3. 异常路径覆盖
  • 4. 状态机完整性（6 项）
  • 5. 跨端一致性（4 项）
  • 6. 术语与表达规范（5 项）
  • 7. 增量检查（1-N 项目，4 项）

评审结论

通过 — 无必须修复项
有条件通过 — 小范围补充
不通过 — 结构性缺失

Phase 6 · 变更同步清单（8 项）

  • 1. requirement.md — 需求稿（始终）
  • 2. changelog.md — 变更日志（始终）
  • 3. review-notes.md — 评审记录
  • 4. PRD 预览 — 重新构建
  • 5. feature-inventory.md — 功能清单
  • 6. 交互稿 + manifest — HTML 原型
  • 7. 截图 — 重新截取
  • 8. 原型对照 — 更新引用

AI 修改需求稿后，会强制逐项走完此清单，确保所有关联产物同步更新。
